Chapter IX

Agents, Tool Use, and Retrieval

A model that can call tools, read documents it doesn't already know, and take actions in an environment is not just a language model — it is the kernel of an operating system. The stack around it is what makes it useful.

Before we start

A language model on its own is a one-shot text predictor. Wrap it in a loop where it can call tools (search the web, read a file, run code, query a database), observe the results, and decide what to do next — and you've built an agent. The pattern is universal: think, act, observe, repeat. The chapter unpacks the modern agent stack.

Two specific components recur. RAG (retrieval-augmented generation) is how an agent reads a corpus it wasn't trained on: turn each document into a vector, find the ones most similar to the user's query, stuff them into the model's context, and ask. MCP (Model Context Protocol) is the open standard introduced in late 2024 for how tools get plugged into models — the "USB for AI tools," more or less. The chapter also covers code agents (the most successful application so far), computer-use models, and the open problems holding the field back.

Terms you'll meet

Agent
A system where a language model decides which actions to take, executes them via tools, observes results, and decides what to do next — in a loop.
Agent harness
The runtime around the model: the loop that runs tools, manages context, applies budgets. Examples: Claude Code, Cursor, OpenHands.
Tool
A function the model can call. Anything from "search the web" to "run this Python" to "send a Slack message."
Tool call
A structured request from the model to invoke a tool with specific arguments — typically a small JSON object.
Function calling
The general API pattern: developers declare tool schemas; the model emits tool calls instead of free text when it wants to use a tool.
Structured output
Model output forced to follow a specific format (JSON schema, regex). The reliability that makes tool calling work.
MCP (Model Context Protocol)
An open standard (Anthropic, November 2024) for how tool servers expose tools, resources, and prompts to language models. By 2026 supported by all major model providers.
ReAct
The "Reason + Act" loop. Interleave reasoning tokens with tool actions until the model emits a final answer. The pattern under almost every production agent.
Reflexion
An agent technique that critiques each finished trajectory in natural language and stores the critique as memory for the next attempt.
Multi-agent
An architecture with multiple LLM instances playing different roles (planner, worker, critic). Useful when subtasks parallelize cleanly.
Toolformer
A 2023 paper that taught a model to insert tool calls inline in its text. The conceptual precursor to function calling.
Retrieval
Finding the most relevant documents (or chunks) from a corpus given a query.
RAG (Retrieval-Augmented Generation)
The standard pattern for grounding a model in a corpus it wasn't trained on: retrieve relevant chunks, stuff them into the prompt, ask the model to answer.
Embedding
A learned vector representation of a piece of text such that semantically similar texts have nearby vectors.
Embedding model
The network that produces embeddings — usually a small transformer fine-tuned with a contrastive loss. Sentence-BERT, OpenAI text-embedding-3, Voyage AI voyage-3.
Cosine similarity
The standard similarity score between two embeddings: their dot product divided by the product of their norms. Ranges from $-1$ to $1$; higher = more similar.
Vector database
A specialized data store for fast approximate-nearest-neighbor search over embeddings. Pinecone, Weaviate, Chroma, Qdrant.
Top-$k$ retrieval
Retrieve the $k$ documents whose embeddings are closest to the query embedding (typically $k = 5$ to $50$).
Dense vs. sparse retrieval
"Dense" = retrieval by embedding similarity. "Sparse" = retrieval by keyword overlap (BM25). Production stacks usually combine both.
BM25
A classic IR scoring function based on weighted keyword overlap. Older than embeddings, still strong for retrieval and used in hybrid stacks.
Reranker / cross-encoder
A second-stage model that scores (query, document) pairs jointly — more accurate than embedding similarity but too expensive for the first pass. Used to re-order the top-$k$ retrieved chunks.
Chunking
Splitting documents into small enough pieces (paragraphs, sections) to fit several into the model's context. The hidden parameter of every RAG system.
Context window
The maximum number of tokens the model can read at once. Modern models reach $10^5$–$10^6$; still a bottleneck for very long documents.
Prompt injection
A security attack: text inside retrieved content or tool output that the model interprets as instructions. The biggest unsolved security problem of 2026 agents.
Computer use
A model class (Anthropic, October 2024 onward) that operates a GUI directly — takes screenshots, plans mouse and keyboard actions, observes the result.
SWE-bench
The dominant benchmark for code agents: real GitHub issues with attached test suites. From <5% solve rate in 2023 to 60–70% by mid-2025.
Code agent
An agent specialized for software tasks. Has a shell, filesystem read/write, test running, and code search. The most successful agent application so far.

§1What "agentic" means, operationally

The word is overloaded. A useful narrow definition: an agent is a system in which a language model decides which actions to take, executes them via tools, observes the results, and decides what to do next — in a loop, with state carried across turns. The model is the planner and decider; tools are the way it touches the outside world.

The minimal contract:

  1. The model is told which tools are available (names, parameter schemas, what they return).
  2. At each step the model either produces a final answer or a tool call: a structured JSON-ish thing naming a tool and its arguments.
  3. The harness executes the tool, returns the result, and feeds it back into the model's context.
  4. Loop until the model emits a final answer or some budget runs out.

This is just an outer loop wrapped around a language model. The novelty is that frontier models from 2024 onward are trained to participate in it — fine-tuned with thousands of tool-use traces so they reliably emit well-formed tool calls, recover from tool errors, and chain tools sensibly. Without that training, an LLM in a tool-use loop falls over fast.

§2Tool use, function calling, and MCP

Earlier work (Toolformer; Schick et al. 2023) showed models could be self-taught to insert tool calls inline in their output text. Production today is more structured. Every frontier API has some flavor of function calling: the developer declares a list of tools with JSON-schema parameter specs; the model produces tool calls as structured outputs (not free text); the harness runs them.

Until late 2024, every developer wired up their own tool integrations against each model's bespoke API. The Model Context Protocol (MCP; Anthropic 2024) standardized this. MCP is an open protocol that defines how tools, resources, and prompts are served to a language model. Tool servers (anything from a filesystem to a Slack instance to a SQL database) expose a uniform interface; LLM clients consume them generically. As of mid-2025 the major model providers all support MCP, and the ecosystem of MCP servers — community and commercial — covers most common integrations.

The practical upshot: a 2026 agent harness composes a model + a set of MCP servers + a conversation loop. No glue code per model, no glue code per tool.

§3ReAct: reason, then act, observe, repeat

The most influential agent paradigm is ReAct (Yao et al. 2023): interleave reasoning tokens with tool actions. A trace looks like

Thought: I need to find the population of Toulouse in 2025.
Action: web_search("Toulouse population 2025")
Observation: [search results …]
Thought: The top result says 511,684. Let me cross-check.
Action: web_search("Toulouse INSEE population 2025")
Observation: [search results …]
Thought: Both agree. Answer: 511,684.
Final answer: 511,684.

ReAct in this exact textual form is mostly historical now — modern models use structured tool calls rather than parsing free text — but the pattern is universal. Almost every production agent is a ReAct loop wearing different clothes: think, act, observe, think, act, observe, until done.

Variants of the pattern address specific failure modes. Reflexion (Shinn et al. 2023) adds a verbal critique step after each trajectory and stores it as memory for the next attempt. Plan-and-execute architectures split planning from execution. Multi-agent systems use one model as a planner and others as workers — useful when subtasks parallelize naturally.

§4Retrieval-Augmented Generation

Most knowledge an agent needs is not in the model's parameters. Retrieval-Augmented Generation (RAG; Lewis et al. 2020) is the standard technique for grounding model outputs in a corpus the model has never seen:

  1. Embed a corpus of documents into vectors with an embedding model.
  2. Given a query, embed it the same way.
  3. Retrieve the top-$k$ corpus documents by similarity (typically cosine similarity, sometimes a learned reranker on top).
  4. Stuff the retrieved chunks into the model's context window along with the query, and ask the model to answer.

The embedding model is the heart of it. Modern embedding models are transformers (often BERT-family encoders or contrastively-trained decoders) fine-tuned to put related text near each other in the embedding space — Sentence-BERT (Reimers & Gurevych 2019) launched the modern recipe; DPR (Karpukhin et al. 2020) extended it to passage retrieval; subsequent work (Contriever, E5, BGE, OpenAI's text-embedding-3, Voyage AI's voyage-3) keeps incrementally improving quality. Production stacks often combine dense retrieval (vectors) with sparse retrieval (BM25) via hybrid scores, and rerank the top 100 with a more expensive cross-encoder.

RAG is the dominant pattern for grounding LLMs in any kind of proprietary or current data — internal documents, recent web pages, codebases. It works well when the corpus has a clean retrieval structure and the relevant chunks are small enough to fit in the context. It works less well when answers require synthesizing many documents or when the relevant chunk is hard to retrieve by surface similarity (the classic "vibe vs. content" failure mode).

§5Code agents

Coding is the agent application that has progressed fastest. The setup is well-defined: the tool is a shell or an IDE, the environment is a code repo, the reward (when training) is whether tests pass. SWE-bench (Jimenez et al. 2023), launched at <5% solve rate by the best 2023 systems, was at 60–70% by mid-2025 for the strongest reasoning-model-driven agents (Claude Sonnet-class with extensive tool use, OpenAI o-series with their coding harness, open-source SWE-agents).

The current generation of production code agents (Claude Code, Cursor, Aider, Codex CLI, OpenHands and similar) gives a model:

  • A working filesystem with read/write/run access.
  • Shell execution.
  • Search across the codebase, semantic and syntactic.
  • An LSP or AST tool for editing.
  • Sometimes: web search, debugger control, version control.

A user task ("refactor this module to use the new API") becomes a multi-step agent trajectory: read the relevant files, draft the change, run the test suite, observe failures, repeat. The model's job is dramatically simpler than "write a correct change from scratch" because it can iterate against feedback.

§6Computer use and the open frontier

Computer use models, debuted by Anthropic in October 2024 and adopted across labs since, can be given a screen and a mouse and asked to operate any GUI: book a flight, fill out a form, navigate a SaaS dashboard. The model takes screenshots, plans mouse/keyboard actions, and observes the result. As of early 2026 these models work reliably on simple tasks, sporadically on complex ones, and reveal the limits of current agent stacks under real-world UI churn.

Three open problems shape the frontier:

  • Long-horizon memory. Context windows of $10^6$ tokens help but are not infinite, and most agents lose track over multi-day tasks. Persistent memory architectures (compress, summarize, retrieve from past sessions) are an active research area.
  • Robustness to adversarial inputs. A web page can include text that a model interprets as instructions ("prompt injection"). Defenses are immature; the security model of agents in 2026 is roughly "assume any tool result might be hostile."
  • Evaluation. Static benchmarks saturate fast. Live evaluation against real, evolving environments — SWE-bench-live, WebArena, OSWorld — is the credible signal, but expensive to run at scale.

The trajectory of agent capability tracks the trajectory of reasoning-model capability from Chapter VIII closely. Most of what has improved between mid-2024 and 2026 is the underlying model; the harness has matured (MCP, better tool design) but has not changed conceptually.
